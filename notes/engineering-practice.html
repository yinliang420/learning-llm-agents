Agent 工程实战 — Eval / Observability / Reliability

从"理论懂"到"能上线"的关键三层工程能力：会测量（Eval）、看得见（Observability）、不轻易挂（Reliability）。这是 agent-lab Phase A→B→C→D+ 的完整记录，所有结论都基于真实数据。

一Eval 框架（Phase A/B）

1.1 为什么 demo 跑通不算"做出来了"

Pattern 01-05 我们各自跑了几个 case 都通过，看起来都"成功"了。但工程上这没意义——单次 demo 不能告诉你：

  • 准确率是 90% 还是 60%？（demo 几次过几次，没统计）
  • 改一行 prompt 后是变好还是变坏？（没有对比基准）
  • 哪类 case 容易失败？（没有分类）
  • 成本 / 延迟分布是怎样？（没有度量）
⚠️ 核心问题

没有 eval，agent 工程就是炼丹——改一改、跑一跑、感觉好一点就发布。每次改动都是赌博。

1.2 框架的核心数据结构

极简但完整的 eval 框架只需要四个数据类。以下是 shared/eval/__init__.py 的设计：

@dataclass
class EvalCase:
    """一个测试用例。"""
    id: str                              # 跨报告对比的唯一 ID
    input: str                            # 喂给 agent 的输入
    expected: dict                        # 期望结果（结构由 scoring_fn 决定）
    tags: list[str]                      # 失败模式分析用
    notes: str                           # 写给自己看：为什么这个 case 存在

@dataclass
class AgentResponse:
    """instrumented agent 返回的完整信息。"""
    text: str
    tool_calls: list[dict]
    cost_usd: float
    duration_ms: int
    turns: int
    finish_reason: str

@dataclass
class RunAttempt:
    """单次执行结果（支持 repeat N 次）。"""
    actual: AgentResponse
    passed: bool
    score: float
    failures: list[str]

@dataclass
class CaseResult:
    """一个 case 的完整结果（含所有 repeat 运行）。"""
    case_id: str
    input: str
    expected: dict
    runs: list[RunAttempt]              # 多次运行结果

    @property
    def pass_rate(self) -> float:
        return sum(1 for r in self.runs if r.passed) / len(self.runs)

    @property
    def flaky(self) -> bool:
        return 0 < self.pass_rate < 1     # 既不是全过也不是全败
设计要点

多次运行（repeat）作为一等公民：单次结果不可信。Pass rate 才是工程指标。

1.3 默认 scorer 的 6 种检查

大多数 agent 评估场景都能用同一个 scorer 函数覆盖。expected dict 支持以下 key：

Key含义例子
contains_all答案必须包含所有子串["7850", "kg/m³"]
contains_any答案至少包含一个["铝", "aluminium"]
not_contains答案不能包含["3422"]（防模型编造数值）
no_tool不应调用任何工具True（概念题）
expected_tool必须调用特定工具"lookup"
min_tools至少调 N 次工具2（对比型问题）
def default_scorer(response: AgentResponse, expected: dict) -> tuple[bool, float, list[str]]:
    failures = []
    text = (response.text or "").lower()
    called = [tc["name"] for tc in response.tool_calls]

    if "contains_all" in expected:
        for s in expected["contains_all"]:
            if s.lower() not in text:
                failures.append(f"missing: '{s}'")

    if expected.get("no_tool") and called:
        failures.append(f"expected NO tool call, got {called}")

    if "expected_tool" in expected:
        if expected["expected_tool"] not in called:
            failures.append(f"expected tool not called: {expected['expected_tool']}")

    # ...其他检查...

    return not failures, 1.0 if not failures else 0.0, failures

对 ex1_materials.py 的 SYSTEM prompt 做一次有依据的改动。

Step 1：跑 baseline

原始 SYSTEM v1（简短）：

SYSTEM_V1 = (
    "You are a materials science assistant. "
    "Always use tools for property data — never recall from memory. "
    "Report values with units."

baseline 结果（10 cases × 1 run）：

Accuracy
90.0%

$0.0029
Avg latency
17.7s
Failed cases

失败的 case：missing_material_tungsten（"钨的熔点是多少？"）
现象：模型调了 lookup → not_found → R2 返回空字符串。

Step 2：假设 + 改动（只改一个变量）

假设：SYSTEM 没明确指导"found:false 时该说什么"。

v2 加入显式 fallback 指令：

SYSTEM_V2 = (
    SYSTEM_V1 +
    # ↓ 新增：明确告诉模型工具失败时该做什么
    "If a tool returns found=false, you MUST: "
    "(1) tell the user the material is not in the database, "
    "(2) list the available materials from the tool's 'available' field, "
    "(3) NEVER guess values when found=false."
)

Step 3：v2 跑 eval，自动对比 baseline

Comparison: baseline (v1) → current (v2) ───────────────────────────────────────── Accuracy : 90.0% → 100.0% +10.0% Cost : $0.0029 → $0.0023 −21% Latency : 17.7s → 11.2s −37% ✅ Improved (1): + missing_material_tungsten
反直觉的发现：加更多指令反而更省更快

v2 加了额外指令，但 cost 降 21%、latency 降 37%。原因是 qwen3.6-plus 是 reasoning model，原 SYSTEM 太模糊，模型要在思考链里推断"我该怎么处理 not found"。思考 token = 真金白银。v2 直接告知，思考链直接短路。

工程教训：Prompt 明确性不只是准确率问题，也是成本问题。

PHASE B1.5 repeat 暴露隐藏的 flake

Phase A 看起来很美好。但单次结果是不可信的——LLM 输出概率性，可能是巧合。

加 repeat=3 重新跑

# shared/eval/__init__.py 加 repeat 参数
def run_eval(agent_fn, dataset, scoring_fn, repeat=1, verbose=True):
    results = []
    for case in dataset:
        case_result = CaseResult(case_id=case.id, ...)
        for attempt in range(repeat):
            response = agent_fn(case.input)
            passed, score, failures = scoring_fn(response, case.expected)
            case_result.runs.append(RunAttempt(response, passed, score, failures))
        results.append(case_result)
    return EvalReport(cases=results, repeat=repeat, ...)

结果：暴露了一个完全不知道的 case

Comparison: baseline (repeat=1) → current (repeat=3) ────────────────────────────────────────────────── 严格 Accuracy : 90.0% → 90.0% +0.0% ← 表面没变 Mean pass rate : 90.0% → 96.7% +6.7% ← 整体确实改善 Flaky cases : 0 → 1 ← ⚠️ 暴露 1 个新问题 ✅ missing_material_tungsten: 0% → 100% (+100%) ← v2 真修复 ⚠️ parallel_density_compare: 100% → 67% (−33%) ← 新暴露 flaky
⚠️ 这是 eval N=1 无法发现的真实缺陷

parallel_density_compare（"SiC 和铝的密度分别是多少，谁更轻？"）跑 3 次：

  • 2 次：模型正确并行调 2 个 lookup → ✓
  • 1 次：模型只调了 1 个 lookup → ✗

有 33% 概率模型"忘记"对比型问题需要查双方。没有 repeat，你会基于一次幸运的全通过得出"agent 完美"的结论，然后被用户 33% 几率的失败打脸。

1.6 Phase A/B 联合的工程教训

阶段你以为的状态实际真实状态
没有 eval"demo 跑通了，应该 OK"完全不知道
eval N=1（baseline）"90% 准确率，1 个失败"误把 flake 当确定性失败
v2 + N=1"改 prompt 后 100% accuracy！"没看到 parallel flake
v2 + N=3"其实 1 个 case 是 flaky"第一次接近真相
两个核心指标
  • 严格 accuracy（全通过 case 的比例）反映"用户能否信赖这个 agent"
  • mean pass rate（每 case pass_rate 的平均）反映"细粒度整体表现"
  • 只看一个会误导，必须同时报告
Cost 随 repeat 线性涨的工程含义

baseline 单次 $0.003，repeat=3 → $0.009。每 run 成本 $0.0003 不变。

eval 的精度有成本，必须按需选 repeat 次数：

  • 开发阶段：repeat=1 快速试
  • 关键决策（要不要发布）：repeat=5-10 高置信度
  • CI 流水线：repeat=3 平衡

二Observability（Phase C）

2.1 没 observability 时的真实痛点

Phase A/B 之后，我们眼下有几个实际遇到但答不上来的问题：

问题 ①：parallel flake 的根因

失败那次和成功那次到底有什么不同？模型为什么决定调 1 个 vs 2 个？思考链里在想什么？
print 出来只是最终 text，完全不知道。

问题 ②：v1→v2 latency 降的细节

17.7s → 11.2s。是 Round 1 思考链短了？还是 Round 2 输出短了？是某些 case 大幅变快，还是平均变快？没有细粒度数据。

问题 ③：成本花在哪里

跑 30 次 eval 总成本 $0.009。哪些 case 最贵？Round 1 vs Round 2 占比？思考链 token vs 输出 token？又是不知道。

这些问题在你转向真实项目（Phase 2）后会变成生产问题。Observability 是从"能跑"到"能维护"的分水岭。

2.2 Trace 的数据模型

Langfuse 把 agent 的一次执行建模成一棵树：

Trace (一次完整的用户交互) ├── Span (任意可计时的操作 — "工具执行" "数据库查询") │ ├── Generation (LLM 调用：messages / output / tokens / cost) │ └── Generation ├── Span │ └── Generation └── Score (附加到 trace 的评分：passed=true、latency_ok=true) User × Session ├── Trace 1 ├── Trace 2 └── Trace N (同一用户、同一会话的多个 trace 聚合)
概念含义对应 agent 里
Trace一次完整的 agent 调用run("问题") 一整次
Generation单次 LLM API 调用shared.llm.call() 一次
Span任何可计时操作工具执行、检索、JSON 解析
Score附加到 trace 的评分eval 给出的 passed
Session多个 trace 的逻辑分组用户多轮对话

Pattern 01 一次执行在 Langfuse 里：

Trace: "钢的密度是多少？" (3.5s, $0.0003) ├─ Generation: Round 1 (1.8s, $0.0001) │ input : [system, user "钢的密度?"] │ output : tool_calls=[{name:lookup, args:{material:"steel"...}}] │ tokens : prompt=452 / completion=89 │ model : qwen-plus │ finish_reason: tool_calls │ ├─ Span: tool execution "lookup" (5ms) │ input : {material:"steel", property:"density"} │ output: {found:true, value:7850, unit:"kg/m³"} │ └─ Generation: Round 2 (1.7s, $0.0002) input : [system, user, asst tool_call, tool result] output : "钢的密度是 7850 kg/m³" tokens : prompt=138 / completion=93 finish_reason: stop
drop-in 集成

Langfuse 提供 OpenAI client 的无缝替换：

if os.getenv("LANGFUSE_PUBLIC_KEY") and os.getenv("LANGFUSE_SECRET_KEY"):
    from langfuse.openai import OpenAI    # 启用：所有 call 自动 trace
else:
    from openai import OpenAI              # 兜底：原行为

整个 shared/llm.py 其他代码、所有 agent 代码、所有 eval 代码都不用改。

2.3 真实例子：用 trace 调试 flake

回到 parallel_density_compare 的 3 个 trace。在 dashboard 里并排打开：

# Trace 1 (✓ 成功)
Generation Round 1:
  output.tool_calls = [
    {name: "lookup", args: {material: "SiC",       property: "density"}},
    {name: "lookup", args: {material: "aluminium", property: "density"}}
  ]
  reasoning_tokens: 320  # 思考链：在想"用户要对比两种材料，应该并行"

# Trace 2 (✗ 失败)
Generation Round 1:
  output.tool_calls = [
    {name: "lookup", args: {material: "SiC", property: "density"}}
    # 只有 1 个！aluminium 没了
  ]
  reasoning_tokens: 180  # 思考链短了一半！

# Trace 3 (✓ 成功)
Generation Round 1:
  output.tool_calls = [...2 个...]
  reasoning_tokens: 340
3 秒内的诊断

Trace 2 的 reasoning_tokens 是 Trace 1/3 的一半（180 vs 320/340）。模型"走捷径"了——看到第一个材料就开始调工具，没把"对比"这个意图算上。

没有 Langfuse：只能 print finish_reason 和 tool_calls 列表，能看到结果不同，但不知道为什么。

2.4 5 大工程问题 → observability 的解

工程问题没 observability有 observability
flaky 的根因只能猜对比 trace 立即看到差异
成本归因看总账单，不知道哪类 case 烧钱dashboard 按 case_id/tag 聚合
延迟瓶颈"整体慢"，不知道哪一步火焰图看到每个 span 耗时
回归检测看总分数trace level diff，哪条变慢/贵
生产事故复盘print 早被覆盖拉历史 trace 完整复盘

成本归因示例

Phase 2 上线后可能出现：

用户 A：1000 次调用 → $5
用户 B：50 次调用 → $40   # 这个用户在干嘛？！
用户 C：200 次调用 → $1

没有 observability，你只看到月底 $50 账单。Langfuse 解法：

from langfuse import get_client
langfuse = get_client()

def run_agent(user_id: str, question: str):
    with langfuse.start_as_current_span(
        name="agent_run",
        input=question,
        metadata={"agent_pattern": "01", "question_type": "lookup"}
    ) as span:
        result = call([{"role":"user","content":question}], tools=TOOLS)
        span.update(user_id=user_id, output=result.text)
        return result

评估与 trace 联动（最强组合）

# 在 run_eval 里
from langfuse import get_client
langfuse = get_client()

for case in cases:
    with langfuse.start_as_current_span(name=f"eval_{case.id}") as span:
        response = agent(case.input)
        passed, score, failures = scoring_fn(response, case.expected)

        # 把 eval 结果作为 score 写回 trace
        langfuse.score_current_trace(
            name="passed", value=1.0 if passed else 0.0,
            comment="; ".join(failures) if failures else "ok"
        )
        langfuse.score_current_trace(
            name="latency_ok", value=1.0 if response.duration_ms < 5000 else 0.0
        )

之后 dashboard 可以：筛"passed=0 的所有 trace"、看 passed 分数随时间变化、把 score 和 metadata 交叉分析。

2.6 生产环境的 4 个关键决策

① Sampling（采样率）

100% trace 在生产会爆配额。策略：

def should_trace(response):
    if not response.passed:           return True  # 失败必留
    if response.cost_usd > 0.05:      return True  # 贵的必留
    return random.random() < 0.1     # 普通的 10% 采样

② Alerting（告警阈值）

  • 单 trace cost > $0.50 → Slack 告警（可能恶意 prompt）
  • 单小时平均 latency 涨 50% → 邮件告警
  • pass rate 跌破 90% → PagerDuty 告警

③ Cost Attribution 必备维度

  • user_id：按用户算账
  • feature：按功能模块算账
  • model：按模型算账
  • pattern：按 agent pattern 算账

④ Privacy / PII 脱敏

def sanitize(messages):
    sanitized = copy.deepcopy(messages)
    for m in sanitized:
        content = m.get("content") or ""
        content = re.sub(r"\b1[3-9]\d{9}\b", "[PHONE]", content)
        content = re.sub(r"\b\d{17}[\dXx]\b", "[ID]", content)
        m["content"] = content
    return sanitized

生产 agent 第一天就该有，不要等出事补。

三Reliability Engineering（Phase D+）

3.1 为什么是下一个必学

你已经看到：LLM 输出是概率性的。但生产环境失败模式比"模型偶尔答错"多得多：

LLM API ├── 网络抖动（连接 reset、DNS 失败） ├── 限流（HTTP 429） ├── 服务端 5xx（DashScope/OpenAI 偶发故障） ├── 超时（reasoning model 卡住） └── 配额耗尽 工具调用 ├── 数据库连接超时 ├── 外部 API 挂了 ├── 工具自己抛异常 └── 返回非预期结构（schema 漂移） 成本控制 ├── 死循环烧光预算 ├── 单次请求 token 爆超 context └── 用户恶意 prompt 触发巨长输出
没有 reliability = 任何组件出问题，整个 agent 链路挂掉

用户拿到 500 错误或空字符串。

3.2 失败的分类——不同失败用不同策略

新手最常见的错误是对所有失败用同一个 except。生产代码必须分类：

类型例子正确策略
Transient（瞬时）网络抖动、限流、5xx重试（指数退避）
Permanent（永久）401、schema 错、bad request快速失败，不重试
Capability（能力）模型答错、工具找不到数据降级 / Fallback
Time（时间）调用太慢超时（cancel 后决定）
Budget（预算）token 爆、cost 超限熔断（直接拒绝）
混用策略 = 灾难
  • 把 401 当 transient 一直 retry → 浪费时间放大问题
  • 把网络抖动当 permanent → 用户白白看到失败
  • 把模型答错当 transient 重试 → 大概率还是错，浪费钱

3.3 五种核心策略 + 实现

策略 ① Retry with Exponential Backoff（最常用）

from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from openai import RateLimitError, APIConnectionError, APITimeoutError

TRANSIENT = (RateLimitError, APIConnectionError, APITimeoutError)

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=30),
    retry=retry_if_exception_type(TRANSIENT),
    reraise=True,
)
def call_with_retry(messages):
    return client.chat.completions.create(...)
⚠️ 陷阱：只能用于幂等操作
  • ✅ LLM 推理：幂等
  • ✅ 查数据库：幂等
  • ❌ 发邮件：非幂等，retry 会发两次！
  • ❌ 写订单：必须加 idempotency_key

策略 ② Fallback Chain（主备模型）

ENDPOINTS = [
    ModelEndpoint("dashscope-primary", primary_client, "qwen-plus",
                  "https://dashscope.aliyuncs.com/compatible-mode/v1"),
    ModelEndpoint("dashscope-backup", backup_client, "qwen-plus",
                  "https://dashscope-intl.aliyuncs.com/compatible-mode/v1"),
    ModelEndpoint("openrouter-fallback", or_client, "qwen/qwen-plus",
                  "https://openrouter.ai/api/v1"),
]

def call_with_fallback(messages, **kwargs):
    last_exc = None
    for endpoint in ENDPOINTS:
        try:
            return endpoint.client.chat.completions.create(
                model=endpoint.model, messages=messages, **kwargs
            )
        except Exception as e:
            last_exc = e
            continue
    raise RuntimeError(f"all endpoints failed: {last_exc}")

关键决策：Schema 兼容（所有 endpoint 支持同 tool schema）、监控 fallback rate（> 5% 要查）。

策略 ③ Timeout（两个层级都要）

# 层级 1：单个 HTTP 请求超时
client = OpenAI(
    api_key=...,
    timeout=30.0,    # 30s 拿不到响应就 cancel
    max_retries=0,   # 关掉 SDK 内置 retry，我们自己管
)

# 层级 2：整个 agent loop 超时（跨线程方案）
from concurrent.futures import ThreadPoolExecutor, TimeoutError as FutureTimeout

def run_agent_with_timeout(question, max_seconds=120):
    with ThreadPoolExecutor(max_workers=1) as executor:
        future = executor.submit(run_agent, question)
        try:
            return future.result(timeout=max_seconds)
        except FutureTimeout:
            return "[超时，请简化问题重试]"

策略 ④ Circuit Breaker（熔断）

连续 N 次失败后直接拒绝调用一段时间，给下游服务恢复时间。

class CircuitBreaker:
    def __init__(self, threshold=5, recovery_seconds=60):
        self.threshold = threshold
        self.recovery = recovery_seconds
        self.failures = 0
        self.opened_at = None

    def call(self, fn, *args, **kwargs):
        if self.opened_at:
            elapsed = (datetime.now() - self.opened_at).seconds
            if elapsed < self.recovery:
                raise RuntimeError(f"circuit OPEN, retry in {self.recovery - elapsed}s")
            self.opened_at = None  # half-open 试一次

        try:
            result = fn(*args, **kwargs)
            self.failures = 0          # 成功 → 计数重置
            return result
        except Exception as e:
            self.failures += 1
            if self.failures >= self.threshold:
                self.opened_at = datetime.now()
            raise
生产经验

每个外部依赖一个独立 breaker（DashScope 一个、向量库一个、外部工具一个），互不影响。

3.5 一个关键的工程权衡：Retry vs Eval Flake

如果我们对 parallel flake（2/3 pass）加 retry：

def agent_with_retry(question):
    for attempt in range(3):
        result = agent(question)
        if eval_passed(result):
            return result
    return result

预期效果：

  • 单次 pass rate 67% → retry 后 ≈ 1 − 0.33³ = 96%

但代价：

  • Latency：失败时 × 3，1/3 case 翻倍
  • Cost：同样 × 3 在失败 case 上
  • 掩盖根本问题：模型不稳定的根因没解决，只是用钱买稳定
用 eval 量化决策
report_no_retry = run_eval(agent, dataset, repeat=5)
report_with_retry = run_eval(agent_with_retry, dataset, repeat=5)

compare_reports(report_no_retry, report_with_retry)

3.6 可靠性的核心监控指标

指标含义告警阈值
first_try_success_rate第一次就成功的比例< 90% 要查
retry_rate触发 retry 的比例> 5% 要查
avg_retries_per_request平均重试次数> 0.5 要查
fallback_rate走 fallback 的比例> 1% 要查
circuit_open_count1 小时内熔断次数> 0 立即告警
p50 / p95 / p99 latency延迟分位数p99 > p50 × 5 有长尾
timeout_rate触发 timeout 的比例> 1% 要查
cost_per_request单请求平均成本突然涨 30% 要查
p99 latency 是最容易被忽视的指标

平均 latency 15s，但 p99 可能 60s，意味着 1% 用户体验是"卡死了"。
