v1Pattern 01

最简单的助手 — Augmented LLM

phase1_patterns/01_augmented_llm/ex1_materials.py
🎯 目标：用户能用自然语言查询材料属性，模型不能猜数据，必须用工具获取。

核心思路

LLM 永远不会执行任何代码。它只是根据你给的 Schema 描述，输出一段 JSON 说"我想调 X 函数，参数是 Y"。你的 Python 代码解析这段 JSON，调真实函数，把结果塞回对话，模型再继续。整个过程固定两轮：

消息流（固定 2 轮）
Round 1 你 → 模型：messages + tool schema 模型 → 你：finish_reason="tool_calls"，输出 JSON 指令 {"name":"lookup","args":{"material":"steel","property":"density"}} 工具执行 你的代码：result = lookup("steel", "density") ← 本地 Python 运行，LLM 看不见 你 → 模型：messages + tool result Round 2 模型 → 你：finish_reason="stop"，输出最终回答

关键代码：工具 Schema（这是模型唯一能看到的东西）

工具定义Python
TOOLS = [{
    "type": "function",
    "function": {
        "name": "lookup",

        # ↓ description 是最重要的字段：告诉模型"什么时候用这个工具"
        # 写 WHEN TO USE，不写 HOW（模型会自己看 parameters）
        "description": "Query a specific property of a material from the database. "
                       "Use when user asks for a CONCRETE VALUE (density, Young's modulus, etc.). "
                       "Never guess — always call this tool for actual data.",

        "parameters": {
            "type": "object",   # 永远是 object：参数必须具名，不能是位置参数
            "properties": {
                "material": {"type": "string", "description": "e.g. 'steel', 'SiC'"},
                "property": {
                    "type": "string",
                    "enum": ["Young's modulus", "density", "melting point", "hardness"]
                    # ↑ enum 是关键：没有 enum，模型会发明 "tensile strength" 等不存在的属性
                }
            },
            "required": ["material", "property"]
            # ↑ required 必须写：否则模型可能漏传参数
        }
    }
}]

关键代码：Agent 核心逻辑（固定 2 轮）

run() 函数Python
def run(question: str) -> str:
    messages = [{"role": "user", "content": question}]

    # Round 1：问模型，附带工具描述（HTTP 请求到 DashScope/Qwen）
    r1 = call(messages, system=SYSTEM, tools=TOOLS, max_tokens=512)

    # 模型直接回答（不需要工具）
    if r1.finish_reason != "tool_calls":
        return r1.text

    # 把模型的 tool_call 回复追加进 messages
    # 注意：必须保留 tool_calls 字段，API 用它匹配后续 tool 结果
    raw = r1.raw.choices[0].message
    messages.append({
        "role": "assistant", "content": raw.content,
        "tool_calls": [tc.model_dump() for tc in (raw.tool_calls or [])]
    })

    # 执行工具（本地 Python），结果写回 messages
    for tu in r1.tool_uses:
        result = DISPATCH[tu["name"]](**tu["input"])
        messages.append({
            "role": "tool",
            "tool_call_id": tu["id"],          # 必须匹配
            "content": json.dumps(result)           # 返回 JSON，不是自然语言
        })

    # Round 2：模型有了真实数据，给出最终回答
    r2 = call(messages, system=SYSTEM, max_tokens=512)
    return r2.text

并行调用是免费的

模型在 Q2 自主并行调了两个工具，你写好 schema，这个优化自动获得。

工具跳过也是信号

Q3 直接 stop（不调工具），说明 description 足够精确让模型判断"不需要"。

错误信息设计

工具返回 available 列表，模型才能告诉用户可以查什么。错误要"可操作"。

JSON vs 自然语言

返回 {"value":200,"unit":"GPa"} 比 "The density is 200 GPa" 让模型提取更可靠。

v2Pattern 02

会循环的助手 — Tool-use Loop

phase1_patterns/02_tool_loop/demo.py
🎯 目标：处理需要多步工具调用的问题，且下一步依赖上一步的结果。

v1 → v2 的关键改动

把固定的 2 轮调用，换成一个 while 循环。模型自己决定什么时候停（finish_reason="stop"）。你的代码只负责：stop 就返回，tool_calls 就继续。

❌ v1（固定 2 轮）
r1 = call(messages, tools=TOOLS)
# 只能处理一轮工具调用
...execute tools...
r2 = call(messages)  # 固定结束
# R2 如果 finish_reason="tool_calls"？
# → r2.text 是空字符串，用户拿到空答案
✅ v2（while 循环）
while turn < MAX_TURNS:
    r = call(messages, tools=TOOLS)
    if r.finish_reason == "stop":
        return r.text  # 任意轮次都可以停
    ...execute tools...
    # 继续循环，直到模型说 stop

完整循环代码

02_tool_loop/demo.py — run()Python
MAX_TURNS = 15   # 安全阀：必须有！没有这行，死循环会持续烧钱

def run(question: str) -> str:
    messages = [{"role": "user", "content": question}]
    turn = 0

    while turn < MAX_TURNS:
        turn += 1
        r = call(messages, system=SYSTEM, tools=TOOLS, max_tokens=1024)

        # 终止条件：模型完成任务
        if r.finish_reason == "stop":
            return r.text

        # 继续条件：模型还需要工具
        if r.finish_reason != "tool_calls":
            return f"[unexpected: {r.finish_reason}]"

        # 历史累积：每轮的 tool_call + 结果都追加进 messages
        raw = r.raw.choices[0].message
        messages.append({
            "role": "assistant", "content": raw.content,
            "tool_calls": [tc.model_dump() for tc in (raw.tool_calls or [])]
        })

        for tu in r.tool_uses:
            result = DISPATCH.get(tu["name"], lambda **_: {"error":"unknown"})(**tu["input"])
            messages.append({"role": "tool", "tool_call_id": tu["id"],
                             "content": json.dumps(result)})
        # → 循环，下一轮可以用上一轮的结果做决策（Pattern 01 做不到这件事）

    return "[max turns reached — check for infinite loop]"

Pattern 02 的真正价值不是"多轮"，是"条件依赖" Turn 2 之所以知道要查哪些材料，是因为 Turn 1 的 list_all() 结果已经在 messages 里了。 Pattern 01 做不到这件事——它只有固定的 2 轮，第一轮的结果无法影响第二轮调哪些工具。

三个踩坑点

坑症状修法
死循环同一工具反复调用，无法停止。
Turn 5: lookup(steel, density) ← 重复		加 MAX_TURNS（必须有）。可选：检测连续相同 tool call。
早停模型只查了 SiC 就 stop，忘了检查其他材料。在 system prompt 明确完成标准："Check ALL materials before concluding."
Context 爆炸轮次太多，messages 过长，模型"忘记"最初的问题。限制 MAX_TURNS，或对历史消息做摘要压缩。
⚠️ 发现问题 → 需要升级到 v3

"什么是杨氏模量？" 这类概念题，v2 也走了完整的循环流程（先考虑是否要调工具，再回答）。
实际上它根本不需要工具，直接从模型知识回答就够了。
更麻烦的是：如果用户问很多这类问题，每个都付了"一次 Tool-use Loop"的成本。

→ 问题：不同类型的问题需要不同的处理路径，应该先分类再派发。
v3Pattern 03

会分流的助手 — Routing

phase1_patterns/03_routing/demo.py
🎯 目标：根据问题类型走不同路径，简单问题不走 loop，整体降成本。

核心思路：先分类，再派发

路由流程
用户问题 ↓ [Router] 便宜模型（max_tokens=10，只输出一个词） → "lookup" 需要查数据库的数值型问题 → "compute" 需要计算的问题（数值已知，做运算） → "general" 概念性问题，直接从知识回答 ↓ [Handler] 对应专家处理器（各有自己的工具集） lookup → v2 的 loop + lookup 工具 compute → v2 的 loop + calculate 工具 general → 无工具，直接调模型

关键代码

03_routing/demo.pyPython
# ── 路由器：便宜模型，只输出一个词 ────────────────────────────────────────
ROUTER_SYSTEM = """Classify into exactly one:
- lookup  : needs material property values from database
- compute : needs arithmetic (values given in question)
- general : conceptual question, answer from knowledge
Reply with ONLY one word."""

def route(question: str) -> str:
    r = call([{"role":"user", "content":question}],
             system=ROUTER_SYSTEM, max_tokens=10)   # max_tokens=10，只输出一个词
    label = r.text.strip().lower()
    if label not in ("lookup","compute","general"):
        label = "general"   # fallback：路由失败就走通用路径
    return label

# ── 主入口：分类 → 派发（关键：传原始 question，不只是 label）──────────────
def run(question: str) -> str:
    label = route(question)
    return HANDLERS[label](question)   # ← 必须传 question，不能只传 label
    # 常见错误：handler(route=label)，handler 不知道用户说了什么

路由的真正价值：成本

简单概念题走 general（0 工具调用），复杂查询走 lookup loop。整体成本降 40-60%。

类别数量控制在 3-5

类别超过 5 个，路由准确率从 ~95% 跌到 ~70%。宁可合并类别，不要细分。

路由本身也会出错

混合型问题（需要 lookup + compute）最容易误分类。要么加 mixed 类别，要么让各 handler 更宽容。

fallback 必须有

路由输出非法值时（如 "计算"）要有 fallback，否则 HANDLERS[label] 会 KeyError。

v4Pattern 04

会规划的助手 — Plan-then-Execute

phase1_patterns/04_plan_execute/demo.py
🎯 目标：先生成完整的 JSON 步骤计划，再批量执行，最后汇总。独立步骤可以并行。

三阶段流程

Phase 1 → 2 → 3
[Phase 1: Planner] 一次 LLM 调用，输出 JSON 计划列表 输入：用户目标 输出：[{"id":1,"material":"SiC","property":"density"}, {"id":2,"material":"steel","property":"density"}, ...] [Phase 2: Executor] 按步骤执行（独立步骤可 asyncio 并行） results[1] = lookup("SiC", "density") → {value:3210} results[2] = lookup("steel", "density") → {value:7850} ... [Phase 3: Synthesizer] 一次 LLM 调用，用所有结果生成最终分析 输入：原始问题 + 所有结果 输出：完整对比报告

关键代码：Planner（输出 JSON，不执行任何操作）

04_plan_execute/demo.py — make_plan()Python
PLANNER_SYSTEM = """You are a research planner.
Given a question, output a JSON plan as a list of lookup steps.
Each step: {"id": int, "material": str, "property": str}
Properties available: Young's modulus, density, melting point, hardness
Output ONLY valid JSON, no explanation."""
# 注意：没有列出工具，Planner 只负责"想"，不负责"做"

def make_plan(question: str) -> list[dict]:
    r = call([{"role":"user","content":f"Question: {question}\nOutput plan as JSON:"}],
             system=PLANNER_SYSTEM, max_tokens=512)
    try:
        text = r.text.strip()
        if text.startswith("```"): text = text.split("```")[1].lstrip("json").strip()
        return json.loads(text)
    except:
        return []

🔴
这是 Plan-then-Execute 最危险的失败模式 Planner 在 Phase 1 就生成了错误参数，Phase 2 盲目执行，4 个步骤全部失败。 ReAct（Pattern 02）遇到失败会自适应调整；Plan-Execute 不会，除非加 Replanner 节点。

修法：在 PLANNER_SYSTEM 加 "Always use English names: steel, aluminium, SiC, copper, titanium"；或在 Phase 2 开始前校验计划合法性。
最大优势：可视化计划

Phase 1 结束后能看到完整计划，可插入 input("Continue?") 人工审查再执行。

并行执行独立步骤

Steps 1-6 互相独立，可用 asyncio 同时执行，速度提升 N 倍。

结构化输出必需

计划必须是 JSON，不能是自然语言，否则 Executor 无法解析执行。

vs Pattern 02

ReAct 自适应强，Plan-Execute 可视化强、可并行。长任务推荐 Plan-Execute + Replanner。

即使 Plan-Execute 成功生成了报告，不同 run 的报告质量波动很大：
有时详细带数据支撑，有时简略缺乏比较。
我们需要一种机制让系统能"检查自己的输出质量，并在不达标时自动改进"。

→ 问题：需要一个独立的评估器来稳定输出质量。
v5Pattern 05

会反思的助手 — Reflection

phase1_patterns/05_reflection/demo.py
🎯 目标：初始输出质量不稳定时，通过独立评估 + 修改循环来稳定质量。

核心原则：Evaluator 必须完全独立于 Generator

⚠️
最重要的一条规则 如果 Generator 和 Evaluator 用同一个 system prompt，模型会给自己打满分（橡皮图章效应）。 Evaluator 必须用不同的 prompt，甚至不同的模型。

关键代码：Generator + Evaluator（两个完全不同的 prompt）

05_reflection/demo.pyPython
MAX_ROUNDS = 2          # 最多修改 2 次（防止无限循环）
QUALITY_THRESHOLD = 8  # 达到 8/10 就停（不追求完美，追求够用）

# ── Generator：生成答案（知道数据，负责输出）────────────────────────────
GENERATOR_SYSTEM = f"""You are a materials science analyst.
Use this data: {DATA_CONTEXT}"""

# ── Evaluator：完全不同的视角（负责挑毛病）────────────────────────────────
# 关键：EVALUATOR_SYSTEM 和 GENERATOR_SYSTEM 完全不同
# 格式要求输出 SCORE/ISSUES/SUGGESTION，方便代码解析
EVALUATOR_SYSTEM = f"""You are a STRICT technical reviewer.
Ground truth data: {DATA_CONTEXT}

Evaluate on: factual accuracy, completeness, clarity.
Respond in this EXACT format:
SCORE: [0-10]
ISSUES: [problems or "none"]
SUGGESTION: [one fix or "none"]"""

# ── Reflection loop ──────────────────────────────────────────────────────
def run(question: str) -> str:
    answer = generate(question)

    for round_num in range(1, MAX_ROUNDS + 1):
        score, issues, suggestion = evaluate(question, answer)

        if score >= QUALITY_THRESHOLD:   # 质量达标，停止（不强制修改）
            break

        # 把 Evaluator 的具体反馈传给 Generator 修改
        answer = generate(question, feedback=f"{issues}. Fix: {suggestion}")

    return answer

何时值得用

只在初始输出质量不稳定时用。如果 10 次 run 中有 3+ 次输出质量差，才考虑加 Reflection。

评分必须结构化

Evaluator 要求输出 SCORE/ISSUES/SUGGESTION。自然语言反馈代码无法解析。

双停止条件

分数阈值（够好就停）+ MAX_ROUNDS（防无限）缺一不可。

越改越错的风险

如果初始答案正确，Evaluator 却挑出"问题"，修改后反而更差。要用 ground truth 评估，不能纯靠主观。

综合优化

工具调用准确率 — 系统性优化指南

以下优化在五个版本中都适用，越早建立越好。

1Description 写 WHEN，不写 HOW

Description 是模型决定是否调用工具的唯一依据。写"这个工具能做什么"比写"这个工具的参数"更重要。

❌ 只写了 HOW（效果差）
"description": "Gets material data by name and property"

# 模型不知道什么时候该用 vs 不用
# 结果：过度调用（概念题也调），或从不调用
✅ 明确写了 WHEN（效果好）
"description": (
  "Use when user asks for a SPECIFIC NUMERIC "
  "property of a named material. "
  "NOT for general conceptual questions. "
  "Never guess — always call this tool."
)
# 模型准确判断何时调（数值查询）
# 何时不调（概念解释）

2有限枚举值必须用 enum

没有 enum 约束，模型会根据上下文"合理推断"参数值，发明数据库里没有的属性名。

❌ 没有 enum
"property": {"type": "string"}

# 模型传了 "tensile strength"
# DB 查不到 → not found error
# 模型不知道有哪些合法属性
✅ 有 enum
"property": {
  "type": "string",
  "enum": ["Young's modulus","density",
           "melting point","hardness"]
}
# 模型只能从合法值里选
# 永远不会传不存在的属性名

3工具返回结构化 JSON，不返回自然语言

模型需要从工具结果中提取数值和单位。自然语言字符串提取不稳定，JSON 提取可靠。

❌ 返回自然语言
return f"The density of {material} is {v} {u}"

# "The density of steel is 7850 kg/m³"
# 模型要从字符串解析 7850
# 偶尔提取错误（如混淆单位）
✅ 返回结构化 JSON
return {
    "found": True,
    "material": mat,
    "value": v,      # 数值类型，不是字符串
    "unit": u
}
# 模型可靠提取 value 和 unit
# 不会混淆或误读

4错误信息要"可操作"，让模型知道下一步

错误发生时，模型需要知道如何继续。空洞的错误信息会让模型卡住或胡乱猜测。

❌ 错误信息不可操作
return {"error": "not found"}

# 模型不知道：
# - 是材料名错了？
# - 是属性名错了？
# - 有哪些可以查？
✅ 错误信息让模型知道怎么办
return {
  "found": False,
  "error": f"'{material}' not in DB",
  "available": list(DB)  # ← 告诉模型有什么
}
# 模型读到后，可以告知用户有哪些材料可查
# 实测：用户问"钨的熔点"，模型优雅回应可用列表

5System Prompt 中加强制规则，减少幻觉

模型有时会绕过工具直接从训练记忆回答（可能是错的）。显式的强制规则能显著提升工具使用率。

system prompt 模板Python
SYSTEM = (
    "You are a materials science assistant. "

    # ↓ 关键：禁止猜测，强制使用工具
    "Always use the lookup tool for property values — never recall from memory. "

    # ↓ 明确什么时候才算完成
    "Check ALL required materials before concluding. "

    # ↓ 输出格式要求
    "Always report values with units."
)

6工具粒度：不能太细，也不能太粗

工具粒度影响模型的调用次数和理解能力。

粒度示例问题
太细 get_density(material)
get_hardness(material)
get_melting_point(material)		 查 3 种属性需要调 3 次，模型需要记住哪些查了哪些没查
合适 lookup(material, property) 一个工具，通过参数区分，模型理解清晰
太粗 get_all_info(material) 返回所有属性，大量 token 浪费在模型不需要的数据上
