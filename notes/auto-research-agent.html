Auto Research Agent — Stage 1 + Stage 2 完整实战记录

从"会用 Claude Code 读论文" → "自己造一个能用的论文 QA agent"的完整工程记录。这份文档不是笔记摘要，是每一个决策点的展开——为什么选这个工具、为什么这样切 chunk、为什么 SYSTEM prompt 加这一句、为什么 Pattern 02 在跨论文场景会到极限。和前两份的关系：基础概念 教你 agent 是什么、工程实战 教你 eval/observability/reliability 怎么做、这一份带你把所学全部用到一个真实研究项目上。

一Auto Research 的设计空间

在动手之前必须先理解："auto research"不是一个产品类目，是一个能力光谱。同一个词，从最简单的"PDF 问答"到最前沿的"自主科学家"，跨越了至少 4 个数量级的复杂度。如果你不先把这个光谱看清楚，会出现一个常见错误：被 Deep Research 那种 demo 吸引，第一步就想做 L3 级别的系统，结果撞上 cost / 引用准确性 / 评估这三座大山，做出来个不能用的玩具就放弃了。

1.1 L1 → L4：能力光谱的四个层级

L1 单文档处理：一篇 PDF/网页 → 总结、提问、翻译、扩展 NotebookLM、ChatPDF、Scholar AI · 完全成熟
L2 文献综述自动化：一组（5-50 篇）文献 → 结构化对比表 / 知识图谱 Elicit、SciSpace、ResearchRabbit · 垂直产品成熟
L3 深度研究助手：开放问题 → 自主上网搜文献 → 综合 → 带引用的长报告 OpenAI/Anthropic/Gemini Deep Research、STORM · 商业可用但局限明显
L4 自主科学家：开放领域 → 提假设 + 跑实验 + 解读结果 + 写论文 + 自我审稿迭代 Sakana AI Scientist、Stanford CoScientist · 研究前沿，尚不可靠

每一级的真实能力边界

L1 — 你今天就能用的

"这篇 NiFe paper 的方法是什么？"、"它的 baseline 是哪些？"、"把这页里的图 3 解释一下"。能做：单文档内的事实抽取、总结、复杂段落解释、关联到训练知识的延伸提问。不能做：跨文档对比（除非你手动喂多个）、检索它没见过的论文、保证引用真实存在（容易编造页码）。

L2 — 我们 Stage 2 在做的

"对比这 5 篇 NiFe paper 的 OER 性能"、"哪几篇用了 hydrothermal 合成？"、"按发表年份排序"。能做：在固定的 corpus 内做结构化对比、找出共性差异、按维度归类。不能做：动态扩展 corpus（你给的就是全部）、保证对比数据的可比性（不同论文用不同电流密度报告 overpotential，agent 不会自动归一化）。

L3 — 商业 Deep Research 的层级

"近 3 年 SiC 在航空结构件上的进展？"、"光催化分解水的最新 SOTA？"。能做：自主规划搜索 query、调外部搜索 API（Google/Tavily/Semantic Scholar）、把找到的 30+ 网页/论文综合成万字报告、自动加引用链接。不能做：保证引用 100% 真实（fabricated 引用是已知问题）、覆盖被 paywall 阻挡的论文、判断老论文是否被新工作 supersede 了。

L4 — 学术前沿，AI Scientist 类

"在这个 sub-field 里提一个有创新性的研究问题，做实验验证，写成论文"。能做：在范围严格定义的封闭领域（如某种 ML 任务）跑通完整闭环、生成可执行代码、写出形式上像论文的产出。不能做：跨学科原创性思考、判断实验设计是否方法学合理、产生真正有学术价值的洞见。Sakana AI Scientist v1 生成的论文里 reviewer agent 给的都是 4-5 分（满分 10），且包含 fabricated 引用。

⚠️ 跳级失败率 95%

L1 → L2 → L3 → L4 是必经之路。每一级都是下一级的子组件：L4 需要 L3 当文献调研器，L3 需要 L2 当综合层，L2 需要 L1 当读单篇的基础。你 PhD 的终极目标是 L4，但学习应该严格按顺序。我们这次做完了 L1 (Stage 1 PDF QA) 和 L2 的基础（Stage 2 RAG 跨论文检索），下一步才是 L3。

1.2 三个必读的参考系统

不需要看遍所有开源项目。读完这三个，你就理解了 80% 的设计空间。每个我都给出它的"核心 trick"——这个系统发明了什么独特的工程模式。

① GPT-Researcher（开源代表，最简单清晰）

GitHub: assafelovic/gpt-researcher

[Planner LLM] 输入：用户研究问题（"近 3 年钙钛矿太阳能电池效率突破？"） 输出：5-10 个子搜索 query 列表 ["highest efficiency perovskite 2023", "perovskite stability advance", "tandem perovskite silicon 2024", ...] ↓ [对每个 query 并行执行] → 调 Tavily/Google 搜索 API → 取前 N 个网页结果 → BeautifulSoup scrape 内容 → 内容 chunk + embed → 存入向量库 ↓ [Synthesizer LLM] → 从向量库按 query 检索 top-k chunks → 生成最终长报告（带引用）
这是什么

最简单的 L3 实现。本质就是 Plan-Execute（Pattern 04）+ RAG。一个周末能读完源码。代码组织清晰，没有花哨的抽象。

核心 trick

把"研究"分解成"很多个并行搜索"。每个 query 独立检索→独立 embed→最后统一综合。没有递归思考，没有反思迭代。简单粗暴但有效。

局限

因为 query 是 Planner 一次性生成的，质量完全取决于 Planner 输出。如果 Planner 漏了某个关键 sub-topic（比如"perovskite degradation mechanism"），最终报告也会漏。不能自适应。

② STORM（学术界代表，结构最有意思）

GitHub: stanford-oval/storm · 论文：NAACL 2024

Pre-writing 阶段（写之前的探索） 1. 给定主题 "钙钛矿光伏" 2. Persona Generator：生成不同视角的"专家" - 材料化学家："关心晶体结构、缺陷工程" - 器件物理学家："关心电荷传输、界面" - 工业工程师："关心稳定性、可量产性" 3. 对每个 persona × 主题，启动 simulated conversation： persona ↔ expert（GPT 扮演领域专家） persona 问问题，expert 回答 + 引用 search 结果 → 每轮对话产生新 references → 加入 reference pool ↓ Writing 阶段（按 outline 强制引用生成） 4. 基于探索结果生成 outline（章节结构） 5. 按 outline 逐节生成 每节强制 cite 已收集的 references 不允许凭空说话，每个 claim 必须有支撑
这是什么

专门做"长格式综述"的系统。论文里跟 Wikipedia 编辑（人类）写的文章对比，发现自动生成的文章在 verifiability、organization 上比直接 prompt GPT-4 显著好。

核心 trick

"Multi-perspective simulated conversation"。不是让 LLM 自己"想这个主题有哪些方面"，而是生成几个不同身份的专家，让他们对话出来。不同 persona 会问出不同子问题，覆盖更广。

为什么这个 trick 重要

它的本质是结构化的"自我多样性"。LLM 单独"自己想"容易卡在一个角度。多个 persona 互相提问，强制覆盖多角度。这个思想可以迁移到任何需要"全面性"的任务。

③ AI Scientist（最前沿，最有野心）

GitHub: SakanaAI/AI-Scientist · v2

[Idea Generator] 输入：基于现有研究领域（如 Diffusion Models） 输出：N 个研究 idea（"explore X variant"） ↓ [Idea Filter] 自动评估 novelty / feasibility，过滤掉不靠谱的 ↓ [Experimenter] → 写代码实现 idea → 跑实验（在准备好的 baseline 上） → 收集 metrics → 如果实验失败，自动 debug + retry ↓ [Writer] → 根据结果写 LaTeX 论文 → 生成图表 → 自动 bibtex 引用 ↓ [Reviewer] → 另一个 agent 扮演审稿人 → 按 NeurIPS rubric 打分 + 给意见 ↓ [Reflector] → 根据 review 改论文（或者推翻重做）
这是什么

试图把整个科研流程自动化。Sakana 用它"生成"了数十篇 ML 论文，每篇成本约 $15。是目前最完整的 L4 尝试。

核心 trick

Pipeline 编排 + 各阶段都用专门的 LLM 角色。Idea/Experiment/Write/Review/Reflect 五个独立 agent，每个有自己的 prompt 和评估标准。

为什么读它的局限性更重要

实际生成的论文里：

  • Fabricated 引用：bibtex 里有不存在的论文（模型记忆错乱生成的）
  • Reviewer 给分集中在 4-5（满分 10）：自己审自己宽容
  • 实验设计常有方法学错误：比如 baseline 选错、metric 不合适
  • "创新"往往是组合现有方法：很少有真正原创洞见

这些不是工程 bug，是 L4 当前能力天花板。读它你会学到：什么问题是 prompt engineering 解决不了的，需要更基础的能力进步。

1.3 L3 级 agent 的 7 个组件 — 与已学技术的映射

把 L3 级 agent 拆开看，里面有 7 个组件。关键洞察：你已经掌握了 90% 的工程零件。学 auto research 不是学新概念，是学怎么组装这些零件来应对研究场景的特殊挑战。

组件做什么对应已学技术具体在我们项目里
Planner 把开放问题分解成可执行 sub-questions / search queries Pattern 04 Plan-Execute Stage 1 没用（Pattern 02 直接 loop），Stage 3 会引入
Searcher 调外部搜索 API 找候选文献/网页 Pattern 01 Augmented LLM 工具调用 Stage 1/2 没用（corpus 固定本地），Stage 3 接 Tavily/arxiv
Reader 把 PDF/HTML 转成 LLM 能处理的文本 Tool design + 长文本截断 + State 外置 Stage 1 的 read_paper(pages)，自动截 12000 字符
Retriever 从已读内容按需检索相关片段 Memory 系统 + State 外置（chunks 在 vector DB） Stage 2 的 rag_search(query)，chromadb + sentence-transformers
Synthesizer 综合多源信息生成最终回答 Plan-Execute 的 Synthesizer 节点 / LLM call 最后一步 Stage 1/2 都是 Round 2 LLM call（融合 tool results 后回答）

💡 这个映射表的工程含义

每次你看到一个新论文/产品演示，先对照这 7 个组件问自己：它怎么实现 Searcher 的？Citation tracker 是怎么防止 hallucination 的？Critic 用了不同模型还是同模型不同 prompt？把答案填进表格，你就在 30 分钟内"理解"了一个新系统。不必读源码。

1.4 研究场景的 4 个特殊工程挑战

通用 agent 工程已经够难（cost、context、eval、reliability 都是问题）。研究场景在这之上还多了 4 个特有挑战，每一个都决定了你的 agent 是否真能"上线"。

① 引用必须 100% 真实 — 学术诚信零容忍

通用 agent 答错一题，用户重试就好。研究 agent 编造一个不存在的论文引用 = 学术诚信问题。如果学生用你的工具写综述，里面有 fabricated 引用，被导师发现后果严重。

为什么 LLM 容易编造引用

LLM 训练数据里见过无数次 bibtex 格式，所以"知道"引用长什么样：作者+年份+期刊+页码。当你让它"写一段带引用的科研内容"，它会从训练记忆里拼凑一个看起来合理的引用——但具体的 paper title、作者列表、年份、卷号、页码可能都是错的，甚至论文根本不存在。这是结构性问题，不是 prompt 调优能根治的。

工程做法（按可靠性递增）：

做法原理可靠性
System prompt 警告"Don't fabricate citations"10-30%（无用）
生成时强制从池里选提前检索一批 references，生成时只能 cite 这些70-90%
独立 verifier agent生成后逐条核对：去 arxiv/semantic-scholar 查 DOI 是否存在95%+
失败的引用直接删段核对失败 → 把这段去掉，宁缺勿滥近 100%
具体实现模式（推荐用法）
def generate_with_verified_citations(question):
    # 1. 提前检索一批候选 references
    refs = search_papers(question, top_k=30)

    # 2. 生成时强制 prompt 只能 cite 这些 ref 的 id
    answer = generate(question, allowed_refs=[r.id for r in refs])

    # 3. 抽取答案里的所有 ref_id，去对应 ref 验证
    cited_ids = extract_citations(answer)
    for ref_id in cited_ids:
        ref = next(r for r in refs if r.id == ref_id)
        if not verify_ref_exists_on_arxiv(ref):
            answer = remove_paragraph_containing(answer, ref_id)

    return answer

② Cost 爆炸 — 研究 = 长 context × 多次调用

一篇英文论文平均 5000-8000 tokens（含 references 可达 15000）。一篇综述任务要读 20-30 篇候选论文，加上多轮规划、合成、修改，单次任务可能烧 $5-50。如果 agent 上线给 1000 用户用，月度账单几万美元起。

你之前学过的所有 cost 控制技术在这里都必须用上：

技术在研究场景的应用预期节省
State 外置（学过）论文全文存 vector DB，messages 里只放 chunks50-80%
Routing（学过）简单分类问题 qwen-turbo，深度综合 qwen-max40-60%
Prompt cachingSystem prompt + tool definitions 缓存20-40%
Batch API大量论文索引 / 摘要任务走 batch 半价50% on batchable tasks
Model tiering检索/分类用 cheap，推理/写作用 strong30-50%
结果 cache同样的论文，多次问相同问题不重复处理视场景

③ 评估极难 — "这是好洞见吗"是主观的

Pattern 01-05 的 eval 可以用 contains_all / expected_tool 自动判分。但研究输出的质量怎么判？"这个综述写得好不好"、"这个 idea 有创新性吗"——没有客观 ground truth。

多层评估策略：

层级评估对象具体做法
可量化子指标引用准确率、覆盖广度、关键 paper 命中率自动核对：cited 引用是否真实存在、答案是否提到了 golden set 里的 ground truth references
结构化检查是否覆盖所有要求的 dimension（如对比方法/数据集/性能）用 schema 约束输出格式，缺字段直接判负
LLM-as-judge整体质量（连贯性、深度、新颖性）用更强的 LLM (GPT-4 / Claude) 按 rubric 打分 1-10
人工标注 golden set"标准答案"你自己写 10 个高质量答案，做语义相似度
Dogfooding真实使用价值把你 PhD 真实需要的某综述让 agent 写，你自己判断好不好
💡 dogfooding 的不可替代性

所有 benchmark 都是 proxy。真正判断研究 agent 是否好用，是看你自己每天会不会用。你 PhD 在读，每天读论文写 idea。让你的 agent 真的服务你的日常研究，6 个月后如果它不能让你的论文写作快 30%，那就是没真用上。

④ 时效性 — 模型 cutoff 是 2025 年初，今天是 2026 年

模型训练截止后的论文，模型完全不知道。即便它信誓旦旦给你介绍"近期进展"，可能都是它编的（或者是它知道的 2024 年内容，假装是 2025 的）。

工程做法：

  • 搜索必须接真实工具：arxiv API、Semantic Scholar API、Google Scholar（注意服务条款）
  • 明确告诉模型 cutoff：system prompt 写 "You don't know papers after [DATE], always search for recent work"
  • Grounding 必须用：Gemini 的 Google Search grounding 或自建 RAG。Anthropic Claude 4.5+ 也有 Web Search beta
  • 时间戳验证：generate 出来的引用，去 arxiv 看 publication date，过老的标 "may be superseded"

2.1 数据准备与采样策略

用户的 PhD 研究藏书：1823 个 PDF 文件 / 7.4GB。从文件名前缀可以推断主要来源期刊：

为什么这种文件名信息有用

看到文件名就知道领域（电催化）+ 期刊 tier（都是高影响力 SCI），不需要打开 PDF 就能粗判藏书质量。如果你的藏书是 arxiv 下载，文件名就是 paper ID，可以直接拼 URL 验证有效性。文件命名规范是元数据的免费来源，导入 RAG 时利用起来能省 embedding 成本。

采样策略：为什么 Stage 1 只用 5 篇

有人会说"既然有 1823 篇，干嘛不全用？"答案是渐进式工程纪律。如果一上来就索引 1823 篇：

  • Embedding 计算耗时长（~20 分钟），出错后重跑成本高
  • Vector DB 测试不易（412 chunks vs 200k+ chunks）
  • Agent 的工具调用日志难分析（信噪比低）
  • eval set 设计困难（不知道里面有什么 paper，怎么写 expected 答案）
  • 第一次架构错误会重写多次，每次都拖慢

正确做法：

采样原则 Stage 1：5 篇（你能完整读过一遍，知道每篇说什么） → 验证架构正确 Stage 2：50-100 篇（够多到必须用 RAG） → 验证检索质量 Stage 3：1823 篇全量 → 验证性能与成本

2.2 三个工具的完整设计

Stage 1 选择 Pattern 02 Tool-use Loop。理由：

  1. 问题"问 PDF" 是典型多步 → 不适合 Pattern 01 单轮
  2. 问题种类不固定 → Pattern 03 Routing 类别太多
  3. 规模小（5 篇）→ Pattern 04 Plan-Execute 杀鸡用牛刀

三个工具的设计经过反复推敲。下面逐个展开。

工具 1：list_papers — 浏览藏书

def list_papers() -> dict:
    """列出所有可用论文，含基本元数据 + 第一页预览。"""
    papers = []
    for pdf in sorted(PAPERS_DIR.glob("*.pdf")):
        try:
            reader = PdfReader(pdf)
            preview = _safe_text(reader.pages[0])
            preview = " ".join(preview.split())[:300]   # 压空白 + 截 300 字符
            papers.append({
                "id": pdf.stem,
                "n_pages": len(reader.pages),
                "first_page_preview": preview,
            })
        except Exception as e:
            papers.append({"id": pdf.stem, "error": f"{type(e).__name__}: {e}"})
    return {"papers": papers, "count": len(papers)}
核心字段

id（文件名 stem，作为 paper 唯一标识）、n_pages（总页数，给模型做 read_paper 时参考）、first_page_preview（前 300 字符，通常足够看到 title + abstract 开头）。

为什么是 300 字符

太短（100）看不到 abstract，模型判断不出主题。太长（1000+）每次 list 都会塞满 messages。300 字符是经验值：够看到 title + 第一句 abstract，又不会爆。这是 token 预算的微观调优。

工具 2：read_paper — 读指定页或前 4 页

def read_paper(paper_id: str, pages: str | None = None) -> dict:
    """读取论文的指定页或前 4 页文本。"""
    pdf_path = PAPERS_DIR / f"{paper_id}.pdf"

    if not pdf_path.exists():
        return {
            "found": False,
            "error": f"Paper '{paper_id}' not found",
            "available_papers": sorted(p.stem for p in PAPERS_DIR.glob("*.pdf")),
        }

    reader = PdfReader(pdf_path)
    total = len(reader.pages)

    if pages is None:
        page_indices = list(range(min(4, total)))   # 默认前 4 页
    else:
        page_indices = _parse_page_range(pages, total)  # 支持 "1-3" / "1,3,5"

    parts = []
    for i in page_indices:
        text = _safe_text(reader.pages[i])
        text = " ".join(text.split())          # 压空白
        parts.append(f"[Page {i+1}]\n{text}")
    full_text = "\n\n".join(parts)

    truncated = False
    if len(full_text) > 12000:                  # MAX_CHARS_PER_READ
        full_text = full_text[:12000]
        truncated = True

    result = {
        "found": True,
        "paper_id": paper_id,
        "total_pages": total,
        "pages_returned": [i + 1 for i in page_indices],
        "text": full_text,
        "char_count": len(full_text),
        "truncated": truncated,
    }
    if truncated:
        last_page = page_indices[-1] + 1
        result["hint"] = (
            f"Text was truncated at 12000 chars. "
            f"Call read_paper again with pages='{last_page + 1}-{min(last_page + 4, total)}' to continue."
        )
    return result
三个关键设计

(1) 默认前 4 页（典型论文 abstract+intro 长度）；(2) 12000 字符硬截断（防 messages 爆炸）；(3) 截断时给 hint 告诉模型怎么继续读。

为什么需要 hint

没有 hint，模型只看到 truncated=true 不知道下一步怎么办，要么放弃要么瞎猜 pages。明确的 hint 让模型知道精确的下一步行动——这是 tool design 的核心原则：错误返回必须 actionable。

工具 3：search_in_paper — 关键词全文搜索

def search_in_paper(paper_id: str, keyword: str, context_chars: int = 200) -> dict:
    """在论文里搜关键词，返回所有匹配位置 + 前后上下文。"""
    # ...路径检查省略...
    keyword_lower = keyword.lower()
    matches = []

    for i, page in enumerate(reader.pages):
        text = _safe_text(page)
        text_lower = text.lower()
        start = 0
        while True:
            pos = text_lower.find(keyword_lower, start)
            if pos == -1: break
            ctx_start = max(0, pos - context_chars)
            ctx_end = min(len(text), pos + len(keyword) + context_chars)
            context = " ".join(text[ctx_start:ctx_end].split())
            matches.append({"page": i + 1, "context": f"...{context}..."})
            start = pos + len(keyword)
            if len([m for m in matches if m["page"] == i + 1]) >= 3:
                break    # 单页最多 3 个匹配

    return {
        "found": True,
        "paper_id": paper_id,
        "keyword": keyword,
        "match_count": len(matches),
        "matches": matches[:20],   # 全文最多 20 个匹配
    }
设计权衡

"单页最多 3 个匹配 + 全文最多 20 个匹配"是双重防爆。某些论文（特别是用 "and" "the" 这种常见词搜）会有几百个匹配，全部返回会塞爆 context。

200 字符上下文怎么定的

太短（50）模型看不出关键词在说什么。太长（500+）每个匹配占用大。200 字符 ≈ 一个英文句子 → 给模型足够语境判断这个匹配是不是相关。同样是 token 预算的微观调优。

2.3 Agent loop 实现

三个工具就位后，agent 本身是 Pattern 02 的标准实现：

MAX_TURNS = 10   # 防死循环

def run(question: str, verbose: bool = True) -> str:
    messages = [{"role": "user", "content": question}]
    turn = 0

    while turn < MAX_TURNS:
        turn += 1
        r = call(messages, system=SYSTEM, tools=TOOLS, max_tokens=1024)

        # 模型说"完成了"
        if r.finish_reason == "stop":
            return r.text

        # 模型要调工具
        if r.finish_reason != "tool_calls":
            return f"[unexpected: {r.finish_reason}]"

        # 把 tool_calls 加进 messages（保留 id 用于配对）
        raw = r.raw.choices[0].message
        messages.append({
            "role": "assistant",
            "content": raw.content,
            "tool_calls": [tc.model_dump() for tc in (raw.tool_calls or [])],
        })

        # 执行每个工具，结果按 tool_call_id 配对回填
        for tu in r.tool_uses:
            name = tu["name"]
            if name not in DISPATCH:
                result = {"error": f"Unknown tool: {name}"}
            else:
                try:
                    result = DISPATCH[name](**tu["input"])
                except Exception as e:
                    result = {"error": f"{type(e).__name__}: {e}"}

            messages.append({
                "role": "tool",
                "tool_call_id": tu["id"],
                "content": json.dumps(result, ensure_ascii=False),
            })

    return "[MAX_TURNS reached]"
Loop 中的 4 个关键工程点
  1. MAX_TURNS = 10：保险栓，没有这一行死循环会一直烧钱。值是经验，简单 QA 任务 5 一般够，跨论文复杂任务可以 15
  2. finish_reason 分支："stop" = 模型完成；"tool_calls" = 模型要工具；其他（length / content_filter）是异常路径，必须处理而不是 silent
  3. tool_calls 必须包含 id：API 要求每个 tool 结果通过 tool_call_id 配对回原始 call，不能漏
  4. 异常隔离：单个工具异常不应该挂掉整个 loop，try/except 把错误返回给模型，让它决定下一步

问题的根源

search_in_paper 只能字符串精确匹配。模型必须想"该用什么关键词"，但作者可能用 "overpotential"、"η"、"ηOER"、"OER η"、"required potential"、"@10 mA cm⁻²" 任一表达。语义相同，字符串不同。模型为了覆盖所有可能性，必须反复尝试。

这不是调 prompt 能解决的，是工具能力的根本限制。需要语义检索而不是字符串匹配——也就是 RAG。

3.3 切分策略 — 字符 vs token、按页 vs 按语义

所有 RAG 系统都要把文档切成 chunks。决策点：

切分单位：字符 vs token

  • 按字符（简单）：固定 N 字符切一刀。优点：无依赖、快。缺点：可能切断单词。
  • 按 token（精确）：用 tokenizer 切到 N tokens。优点：能精确控制 embedding 输入长度。缺点：需要加载 tokenizer，慢。

切分边界：按页 vs 按语义

  • 按页（论文友好）：PDF 一页 ~ 3000 字符，天然就是一个段落上下文。优点：保留原文结构。缺点：跨页内容会丢边界。
  • 按语义段落（NLP 友好）：检测段落/句子边界。优点：每个 chunk 是完整语义单位。缺点：实现复杂、对 PDF 提取质量敏感。
  • 固定窗口 + 重叠（简单且鲁棒）：每 N 字符切一刀，相邻 chunk 重叠 M 字符。优点：简单粗暴有效。缺点：会切断内容但重叠能缓解。
我们的选择：按页内 + 固定 800 字符窗口 + 200 字符重叠

具体逻辑：先把 PDF 按页提取，再对每页的文本做 800 字符窗口 + 200 字符重叠的切分。这样兼顾了：

  • 页边界保留：跨页的内容会被分到不同 chunks，但每个 chunk 知道自己来自哪页（保 page metadata）
  • 边界缓解：相邻 chunks 重叠 200 字符 ≈ 1 句话，被切断的内容总有一个 chunk 完整包含
  • embedding 长度：800 字符 ≈ 200 tokens，远小于 MiniLM 的 256 token 上限

chunk 大小的经验值

chunk 大小检索粒度问题类型适合
200-400 字符很细，单句级FAQ / 事实查询（"作者是谁"）
500-1000 字符段落级论文 QA、技术文档
1500-2000 字符多段落长文档综合理解
3000+ 字符整页 / 整章需要完整上下文才能理解的复杂内容

3.4 Indexer 完整剖析

indexer 做的事：扫描 PDF → 切 chunks → 转 embedding → 入库。下面是完整代码 + 每段的工程决策。

1. 全局配置

2. Chunk 切分

def chunk_page_text(text: str, chunk_size: int = CHUNK_CHAR_SIZE,
                    overlap: int = CHUNK_OVERLAP) -> list[str]:
    """把一页文本切成多个有重叠的 chunk。"""
    text = " ".join(text.split())   # 压缩空白（连续空格/换行 → 单空格）
    if len(text) <= chunk_size:
        return [text] if text else []

    chunks: list[str] = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        if end >= len(text):
            break
        start = end - overlap   # 重叠：下个 chunk 起点回退 overlap
    return chunks
边界处理

三个 edge case：(1) 空文本返回 []；(2) 文本短于 chunk_size 返回单元素列表；(3) 最后一个 chunk 长度可能 < chunk_size，不补 pad。

压缩空白的重要性

PDF 提取的文本经常有连续换行、多空格、tab 等。直接 embed 会污染向量。" ".join(text.split()) 是 Python 惯用法，一行搞定所有空白压缩。

3. PDF → chunks（带 metadata）

def extract_chunks_from_pdf(pdf_path: Path) -> list[dict]:
    """单 PDF → list of chunk dicts。"""
    reader = PdfReader(pdf_path)
    paper_id = pdf_path.stem
    result = []
    for page_num, page in enumerate(reader.pages, start=1):
        try:
            text = page.extract_text() or ""
        except Exception:
            continue      # 某页损坏不影响其他页
        for i, chunk in enumerate(chunk_page_text(text)):
            if len(chunk.strip()) < 50:   # 过滤太短的 chunk
                continue
            result.append({
                "paper_id": paper_id,
                "page": page_num,
                "chunk_idx": i,
                "text": chunk,
            })
    return result
3 个关键设计点
  • 每页独立 try/except：PDF 经常有个别页损坏，不能一页坏全文废。
  • 过滤 < 50 字符的 chunk：这类通常是页眉/页脚噪声，embed 没价值还污染检索。
  • metadata 完整：paper_id + page + chunk_idx 三元组唯一确定一个 chunk。page 用于后期"在第几页找到的"展示。

4. 向量库 + 嵌入入库

def get_collection(reset: bool = False):
    """打开/创建 chromadb collection。"""
    client = chromadb.PersistentClient(
        path=str(DB_PATH),
        settings=Settings(anonymized_telemetry=False),  # 关掉默认 telemetry
    )
    if reset:
        try:
            client.delete_collection(COLLECTION_NAME)
        except Exception:
            pass
    return client.get_or_create_collection(
        name=COLLECTION_NAME,
        metadata={"hnsw:space": "cosine"},   # 余弦相似度（默认 L2）
    )


def index_papers(papers_dir: Path, reset: bool = False) -> dict:
    pdfs = sorted(papers_dir.glob("*.pdf"))
    encoder = SentenceTransformer(EMBEDDING_MODEL)
    collection = get_collection(reset=reset)

    for pdf in pdfs:
        chunks = extract_chunks_from_pdf(pdf)
        if not chunks:
            continue

        # 批量 embed（比逐条快很多）
        texts = [c["text"] for c in chunks]
        embeddings = encoder.encode(texts, show_progress_bar=False, convert_to_numpy=True)

        # 拼 ids 和 metadata
        ids = [f"{c['paper_id']}__p{c['page']}_c{c['chunk_idx']}" for c in chunks]
        metadatas = [{"paper_id": c["paper_id"], "page": c["page"],
                      "chunk_idx": c["chunk_idx"]} for c in chunks]

        # upsert：可重复运行不会出错
        collection.upsert(
            ids=ids,
            embeddings=embeddings.tolist(),
            documents=texts,
            metadatas=metadatas,
        )
4 个工程亮点

(1) 批量 encode：sentence-transformers 一次 encode 一批比逐个快 5-10×；(2) cosine 而非默认 L2：对归一化后的 embedding cosine 更稳；(3) upsert 而非 insert：允许重复运行 indexer；(4) id 设计：{paper_id}__p{page}_c{idx} 唯一且可读。

为什么是 upsert

开发过程中你会反复改 chunking 参数、重跑 indexer。如果用 insert，重复 id 会报错；如果先 delete 全部再 insert，每次都浪费时间。upsert 是开发友好的选择——同 id 覆盖，不同 id 新增。

3.5 Retriever 完整剖析

indexer 是一次性的，retriever 是每次查询都要用的。完整代码：

# 单例 — 第一次 import 加载，之后复用
_encoder: SentenceTransformer | None = None
_collection = None


def _get_encoder() -> SentenceTransformer:
    global _encoder
    if _encoder is None:
        _encoder = SentenceTransformer(EMBEDDING_MODEL)
    return _encoder


def _get_collection():
    global _collection
    if _collection is None:
        client = chromadb.PersistentClient(path=str(DB_PATH), settings=Settings(anonymized_telemetry=False))
        _collection = client.get_collection(name=COLLECTION_NAME)
    return _collection


def rag_search(query: str, paper_ids: list[str] | None = None, top_k: int = 5) -> dict:
    """语义检索向量库。"""
    try:
        col = _get_collection()
    except Exception as e:
        return {"found": False, "error": f"DB not ready: {e}. Run indexer.py first."}

    encoder = _get_encoder()
    query_vec = encoder.encode([query], show_progress_bar=False).tolist()[0]

    # 构造 chroma 的 where 过滤
    where = None
    if paper_ids:
        if len(paper_ids) == 1:
            where = {"paper_id": paper_ids[0]}
        else:
            where = {"paper_id": {"$in": paper_ids}}

    try:
        res = col.query(
            query_embeddings=[query_vec],
            n_results=top_k,
            where=where,
            include=["documents", "metadatas", "distances"],
        )
    except Exception as e:
        return {"found": False, "error": f"Query failed: {e}"}

    docs = res["documents"][0] if res["documents"] else []
    metas = res["metadatas"][0] if res["metadatas"] else []
    dists = res["distances"][0] if res["distances"] else []

    matches = []
    for doc, meta, dist in zip(docs, metas, dists):
        text_compact = doc[:400] + ("..." if len(doc) > 400 else "")
        matches.append({
            "paper_id": meta["paper_id"],
            "page": meta["page"],
            "text": text_compact,
            "similarity": round(1.0 - dist, 3),
        })

    return {
        "found": True,
        "query": query,
        "match_count": len(matches),
        "matches": matches,
        "filter_paper_ids": paper_ids,
        "hint": (
            "These are semantically-retrieved chunks. Use them as-is — "
            "do not re-search with rephrased queries unless similarity is very low."
        ),
    }

逐段工程剖析

单例 encoder + collection

SentenceTransformer 加载需要 ~2 秒，每次 rag_search 都加载会非常慢。用 module-level 单例，第一次 init 后复用。

为什么不用类

函数式更适合作为 tool，调用方不需要 instantiate 一个 Retriever 对象。单例隐藏在 module 内部。

where 过滤的两种写法

chroma 对单值用 {"paper_id": "X"}，对多值用 {"paper_id": {"$in": [...]}}。两个 case 都要处理，否则 single paper 查询会失败。

为什么允许 paper_ids 限定

有时 agent 知道答案肯定在某 1-2 篇里，不需要全局搜。限定范围能显著提升信噪比——你不会被无关论文的"看起来相关"的 chunk 干扰。

chunk 截断到 400 字符

原 chunk 800 字符，返回时截到 400 + "..."。

为什么截

State 外置思想的延伸。完整 chunk 存在 chromadb 里（state），messages 里只放精简版。如果返回 top_k=5 都是 800 字符，单次 tool result 就是 4000+ 字符，loop 几轮就爆 context。

截到 400 仍然够看出大意，模型可以决定要不要 read_paper 看全文。

cosine distance → similarity

similarity = 1 - distance。chromadb 返回 distance（越小越相似），人类习惯 similarity（越大越相似）。

为什么 round 到 3 位小数

similarity 是浮点数。原始 0.5783214... 太长，3 位小数 0.578 够看趋势。减少 token 占用——20 个 matches 每个少几位小数，累计也是几十 token 节省。

hint 字段

返回时附带一句话："These are semantically retrieved chunks. Use them as-is — do not re-search..."

为什么 hint 重要

第一次跑 stage 2 时遇到 agent 反复用不同 query 搜同样意图。加 hint 后，模型在 tool result 里直接看到"不要 re-search"，行为收敛快很多。把 tool 当成"既能给数据，也能给指引"的接口。

深度分析：为什么最贵的 case 反而更贵

hard_oer_overpotential_compare 在 Stage 2 反而 cost +19%。原因：

  • RAG 拿到的 chunks 经常包含 "overpotential at X mA/cm²" 但不同论文 X 不同（10 vs 100）
  • 模型为了"对齐对比"必须反复确认每篇用的是哪个电流密度
  • 这本质是数据结构问题，不是检索问题——RAG 解决"找到内容"，解决不了"内容格式不统一"

根治需要 Plan-Execute Pattern 04——planner 先决定"每篇用什么电流密度报告的"，再去针对性检索。或结构化抽取——先把每篇的 overpotential 数据按统一 schema 抽出来存到结构化字段，再对比。

四教训、模式与下一步

4.1 五个真实的工程教训

教训 ① 整体改进不明显 ≠ 没改进

Stage 2 整体 latency 只降了 25%，cost 几乎一样。粗看似乎"RAG 没用"。但单 case 看，有些 −46%、有些 +19%，差异巨大。整体平均掩盖了细节信号。

工程动作：永远看 case-by-case 的对比，而不只是 mean。Pareto 改进（部分变好，部分变坏）也是改进，关键看变好的是不是更重要的 case。

教训 ② RAG 不是银弹

引入新工具盲目期待"全面 carry"是新手心态。RAG 只对"需要精确语义匹配"的任务有显著帮助。元数据问题（count / list / longest）list_papers 已经够用，RAG 反而是浪费。

工程动作：每次引入新工具，先问"这工具不适合什么场景"，确保 prompt 引导模型在合适场景才用它。

教训 ③ 最难的 case 暴露架构极限

cross-paper 数字对比在 Stage 1 和 Stage 2 都卡。Pattern 02 + RAG 不能根治。要根治必须升级架构（Pattern 04 Plan-Execute）。

工程动作：当 eval 里某个 case 怎么调都过不去（或代价巨大），说明架构到极限了，不是 prompt 问题。这时候才该升级 Pattern。

教训 ④ 新工具上线第一次往往退化

Stage 2 第一次跑反而比 Stage 1 贵 70%、撞 MAX_TURNS。这不是 RAG 的问题，是没针对新工具调整 prompt 和返回格式。

工程动作："上工具 + 改 SYSTEM + 调返回格式" 是一个 atomic 改动，三件事必须一起做。只加工具不改 SYSTEM，模型不知道怎么用；只改 SYSTEM 不调返回格式，messages 会爆炸。

教训 ⑤ eval 设计反模式会反复出现

这 4 天工作里，我们 3 次遇到 eval 关键词列表太严、规则太死的问题：

  • bifunctional_role：expected_tool 反模式
  • missing_paper × 2 次（不同模型，都因 "无法/可用" 不在关键词列表里挂了）

工程动作：生产级 eval 必须用 LLM-as-judge，不能光靠 contains_any。或者每次跑都人工 review 5-10 个抽样案例，发现 eval 漏判及时补关键词。

E.2 升级到 Pattern 04 Plan-Execute — 最大改进，1-2 周

问题：cross-paper 数字对比 cost 不可控（hard_oer_overpotential_compare 始终 $0.01+）。

修法：重写 agent 为 plan-execute 架构

def plan_execute_agent(question):
    # Phase 1: Planner 出结构化计划
    plan = planner_llm(question)
    # plan = [
    #   {action: "extract", paper: "cssc",      field: "OER_overpotential_at_10mA"},
    #   {action: "extract", paper: "ijhydene",  field: "OER_overpotential_at_10mA"},
    #   {action: "extract", paper: "catal8080328", field: "OER_overpotential_at_10mA"},
    #   {action: "compare", inputs: [1,2,3], criterion: "min"},
    # ]

    # Phase 2: Executor 按 plan 执行
    results = {}
    for step in plan:
        results[step.id] = execute_step(step, results)

    # Phase 3: Synthesizer 生成最终答案
    return synthesizer_llm(question, results)

预期：解决 cross-paper 死循环问题。风险：引入"计划错误级联"（我们 conceptual-qa 里学过）；planner 出错会全错。

E.3 扩规模索引到 1823 篇 — 真正的 RAG 主战场

当前状态：5 篇索引 / 412 chunks。Stage 1 的 search_in_paper 在 5 篇上还能跑，但在 1823 篇上根本不可能。这才是 RAG 真正展示价值的场景。

步骤：

  1. 运行 uv run python indexer.py --dir papers --reset
  2. 预计 ~15-20 分钟（CPU embedding 1823 篇 × 80 chunks ≈ 150k chunks）
  3. chromadb 文件大约 500MB-1GB
  4. 加几个"找针对的论文"的 case（如"找所有 2024 年后用 hydrothermal 合成 NiFe 的论文"）
  5. 看 RAG 在"从 1823 候选里找针对的"上的表现

预期：完成后 paper_qa 就真的对 PhD 有用了，每天读文献时可以用。风险：低，是规模扩展。

E.4 加本地 trace logger — 自建轻量 observability

不用 Langfuse 云服务，5 分钟搭一个本地 JSON 文件 trace logger：

@contextmanager
def trace_run(question):
    trace = {"question": question, "started": time.time(), "calls": []}
    yield trace
    trace["duration"] = time.time() - trace["started"]
    Path("traces").mkdir(exist_ok=True)
    Path(f"traces/{trace['started']}.json").write_text(json.dumps(trace, indent=2))

# 在 call() 里
trace["calls"].append({"messages": ..., "response": ..., "cost": ...})

之后用 jq 或写个简单的 viewer.py 就能回放任意一次 run。价值：等真撞到诊断难题（"为什么这个 case 偶尔会失败"）时不用从头重跑，可以从历史 trace 里找。
